Modelar Storage Modes — Import, DirectQuery, Dual 🔥 muy frecuente
Lo usás sin saber cómo se llama cada uno
▶

Import: Los datos se copian al modelo de Power BI. Máxima performance, ideal para datasets que no cambian constantemente. Permite todas las funciones DAX.

DirectQuery: Power BI consulta la fuente en tiempo real, no almacena datos. Siempre actualizado, pero más lento y con limitaciones DAX.

Dual: La tabla puede actuar como Import O DirectQuery según el contexto de la consulta. Ideal para tablas de dimensiones en modelos mixtos.

Regla del examen: Si preguntan "near real time" → DirectQuery. Si preguntan "mejor performance" → Import. Dimensiones en modelo mixto → Dual.
Truco de memoria: DirectQuery = Direct al origen = siempre fresco pero más lento.
Datos Query Folding 🔥 muy frecuente
El secreto de performance en Power Query
▶

Query Folding es cuando Power Query traduce las transformaciones que hacés en el editor a código nativo de la fuente (SQL, OData, etc.) y deja que la fuente haga el trabajo pesado.

Si query folding ocurre → la fuente filtra, ordena y agrupa ANTES de enviar datos → menos datos transferidos → más rápido.

Si no ocurre → Power BI descarga todo y procesa localmente → lento con datasets grandes.

Cómo verlo: En Power Query, clic derecho en un paso → "View Native Query". Si está disponible → hay query folding. Si está en gris → no hay folding en ese paso.
Rompe el folding: Agregar columnas personalizadas (M code), combinar fuentes distintas, ciertos tipos de transformaciones. Los pasos DESPUÉS de uno que rompe el folding tampoco tienen folding.
Gestionar Row-Level Security (RLS) 🔥 muy frecuente
Seguridad de datos por usuario
▶

RLS restringe qué filas de datos ve cada usuario. Hay dos tipos:

Static RLS: Los roles tienen filtros fijos. Ej: Vendedor Argentina solo ve filas donde País = "Argentina". Se configura en Power BI Desktop.

Dynamic RLS: Los filtros usan la función USERNAME() o USERPRINCIPALNAME() para comparar con una tabla de seguridad. Un solo rol sirve para todos.

FILTER(Tabla, Tabla[Email] = USERPRINCIPALNAME())
Flujo: Crear rol en Desktop → publicar → en Service asignar usuarios al rol. Los admins del workspace ven TODOS los datos sin importar el RLS.
Examen: Static RLS → múltiples roles, más fácil de configurar. Dynamic RLS → un solo rol, escala mejor, usa USERNAME()/USERPRINCIPALNAME().
Modelar Esquema estrella vs copo de nieve ⚡ frecuente
La base de todo modelo correcto
▶

Esquema estrella: Una tabla de hechos central (ventas, transacciones) conectada a tablas de dimensiones desnormalizadas (clientes, productos, fechas). Power BI optimizado para esto.

Copo de nieve: Las dimensiones se normalizan en sub-dimensiones. Ej: Producto → Subcategoría → Categoría como tablas separadas.

Siempre elegí estrella en el examen a menos que pregunten específicamente por normalización. Estrella = mejor performance en Power BI.

Tabla de hechos: Contiene medidas numéricas y claves foráneas. Granularidad = un registro por transacción.

Tabla de dimensiones: Contiene atributos descriptivos. Sin duplicados en la clave primaria.

Modelar Relaciones: dirección del filtro cruzado 🔥 muy frecuente
Por qué algunos filtros no funcionan como esperás
▶

Single (una dirección): El filtro va de la tabla "1" a la tabla "muchos". Dimensión → Hechos. Es el default y el recomendado.

Both (bidireccional): Los filtros fluyen en ambas direcciones. Útil para tablas puente en muchos-a-muchos, pero puede causar ambigüedad.

Cuándo usar Both: Solo cuando sea necesario para tablas intermedias (puente) en relaciones muchos-a-muchos. Evitarlo en general — puede causar resultados inesperados.
Regla: Si la relación es 1-a-muchos → Single. Si hay tabla puente en N-a-N → Both o usar CROSSFILTER en DAX.
Datos Tipos de conectores y modos de conexión ⚡ frecuente
Get data: ¿desde dónde y cómo?
▶

Fuentes de datos principales: SQL Server, Azure SQL, Excel, SharePoint, Dataverse, Power BI datasets, Dataflows, Web, API REST.

On-premises data gateway: Necesario cuando la fuente está en la red local (SQL Server on-prem, archivos locales) y necesitás actualizar desde el Service. Hay dos modos: Standard (compartido, múltiples usuarios) y Personal (solo un usuario, no recomendado para empresas).

Examen: ¿Cuándo necesitás gateway? Cuando la fuente de datos es on-premises y el reporte está en Power BI Service. Sin gateway, el refresh falla.
Visualizar Filtros: tipos y jerarquía ⚡ frecuente
Visual, página, informe y drill-through
▶

Jerarquía de filtros (de mayor a menor alcance):

1. Filtro de informe: Afecta todas las páginas.

2. Filtro de página: Afecta todos los visuales de esa página.

3. Filtro de visual: Afecta solo ese visual.

4. Drill-through: Filtra una página de detalle según el contexto del elemento seleccionado.

Slicer vs Filtro: Los slicers son visuales interactivos. Los filtros en el panel Filtros pueden ocultarse del usuario final. Para experiencia del usuario → slicer. Para control que el usuario no debe cambiar → panel de filtros.
Gestionar Workspaces y licencias ⚡ frecuente
Free vs Pro vs Premium
▶

Free: Solo "Mi área de trabajo". No puede compartir con otros (excepto en capacidad Premium).

Pro: Acceso a workspaces colaborativos, compartir contenido con otros usuarios Pro. Para publicar y consumir en workspace compartido necesitás Pro (o Premium).

Premium (por capacidad): Usuarios Free pueden consumir contenido de workspaces Premium sin licencia Pro propia. También habilita Paginated Reports, AI features, mayor capacidad de datos.

Regla del examen: Usuario A (Pro) quiere compartir reporte con Usuario B → B necesita Pro también, a menos que el workspace esté en capacidad Premium.
Visualizar Bookmarks y botones de navegación ⚡ frecuente
Interactividad avanzada en reportes
▶

Bookmark: Captura el estado del reporte en un momento dado: filtros activos, qué visuales son visibles, selecciones. Sirve para crear navegación, tours guiados o estados alternativos del reporte.

Botones de acción: Pueden ejecutar bookmarks, navegar a páginas, URLs, drill-through o Q&A. Se configuran en el panel Format → Action.

Caso de examen típico: Crear un panel de "vista ejecutiva" y otro de "vista detalle" usando bookmarks que muestran/ocultan visuales. El botón toggle entre estados.
Datos Dataflows y Dataverse ⚡ frecuente
Reutilización de transformaciones
▶

Dataflows: Transformaciones Power Query creadas y almacenadas en el Service. Múltiples reportes pueden reutilizar el mismo dataflow. Reduce duplicación de lógica de limpieza.

Dataverse: Base de datos en la nube (antes Common Data Service). Se conecta con Power Apps y Power Automate. Para reportar sobre apps de negocio.

Cuándo usar Dataflow: Misma transformación compleja usada en varios reportes. Centraliza la lógica, equipo de datos mantiene un solo lugar.

DAX Cheatsheet 🔢

Las funciones que más aparecen en el examen. Con ejemplos prácticos.

CALCULATE — La función más importante del examen
Evalúa una expresión en un contexto de filtro modificado
▶

Sintaxis:

CALCULATE(expresión, filtro1, filtro2, ...)

Ejemplos:

-- Ventas solo de Argentina Ventas AR = CALCULATE(SUM(Ventas[Monto]), Ventas[País] = "Argentina") -- Ventas del año pasado Ventas LY = CALCULATE([Total Ventas], SAMEPERIODLASTYEAR(Fechas[Fecha])) -- Quitar filtro de categoría Ventas Todas Cat = CALCULATE([Total Ventas], ALL(Productos[Categoría]))
Clave: CALCULATE puede modificar, agregar o quitar filtros del contexto. ALL() dentro de CALCULATE quita filtros de esa columna/tabla.
Funciones de Time Intelligence
YTD, MTD, QTD y comparaciones con períodos anteriores
▶
-- Año hasta la fecha Ventas YTD = TOTALYTD([Total Ventas], Fechas[Fecha]) -- Mes hasta la fecha Ventas MTD = TOTALMTD([Total Ventas], Fechas[Fecha]) -- Mismo período año anterior Ventas PY = CALCULATE([Total Ventas], SAMEPERIODLASTYEAR(Fechas[Fecha])) -- Mover N períodos Ventas -3M = CALCULATE([Total Ventas], DATEADD(Fechas[Fecha], -3, MONTH))
Requisito: Las funciones Time Intelligence necesitan una tabla de fechas marcada como "tabla de fechas" con una columna de tipo Date sin huecos.
Examen: Si preguntan "ventas del año pasado mismo período" → SAMEPERIODLASTYEAR. Si preguntan "acumulado del año" → TOTALYTD.
FILTER, ALL, ALLEXCEPT
Control del contexto de filtro
▶
-- FILTER: retorna tabla filtrada Ventas Premium = CALCULATE( [Total Ventas], FILTER(Productos, Productos[Tier] = "Premium") ) -- ALL: quita filtros de toda la tabla o columna % del Total = DIVIDE([Total Ventas], CALCULATE([Total Ventas], ALL(Ventas))) -- ALLEXCEPT: quita todos los filtros EXCEPTO los de las columnas especificadas Ventas País = CALCULATE([Total Ventas], ALLEXCEPT(Ventas, Ventas[País]))
Diferencia clave: ALL(tabla) quita todos los filtros de la tabla. ALLEXCEPT(tabla, col) quita todos excepto los de esa columna.
RELATED, RELATEDTABLE, LOOKUPVALUE
Navegar relaciones entre tablas
▶
-- RELATED: trae columna de tabla relacionada (lado 1) Categoría = RELATED(Productos[Categoría]) -- en tabla de hechos -- RELATEDTABLE: retorna tabla de relación (lado muchos) CantProd = COUNTROWS(RELATEDTABLE(Productos)) -- en tabla dimensión -- LOOKUPVALUE: busca sin relación formal Ciudad = LOOKUPVALUE(Ciudades[Nombre], Ciudades[ID], Ventas[CiudadID])
Cuándo usar LOOKUPVALUE: Cuando no podés crear una relación formal pero necesitás traer un valor de otra tabla.
SUMX, AVERAGEX, COUNTX — Iteradores
Evalúan expresión fila por fila
▶
-- SUMX itera la tabla y suma la expresión por fila Ingresos = SUMX(Ventas, Ventas[Cantidad] * Ventas[PrecioUnitario]) -- Equivale a una columna calculada, pero sin crearla -- Útil cuando no tenés la columna de ingresos precalculada
SUM vs SUMX: SUM suma una columna existente. SUMX calcula una expresión fila por fila y luego suma. SUMX es más flexible pero más lento en tablas grandes.
Variables en DAX (VAR)
Hacen el código más legible y eficiente
▶
Variación % = VAR VentasActual = [Total Ventas] VAR VentasAnterior = CALCULATE([Total Ventas], SAMEPERIODLASTYEAR(Fechas[Fecha])) VAR Variación = DIVIDE(VentasActual - VentasAnterior, VentasAnterior) RETURN IF(ISBLANK(VentasAnterior), BLANK(), Variación)
Ventajas VAR: Se calculan una vez (no múltiples veces). El código es más legible. Permite usar RETURN para controlar el resultado final.

🆚 DP-700 vs PL-300 — ¿en qué se diferencia?

El PL-300 es sobre consumir y visualizar datos (Power BI). El DP-700 es sobre construir la infraestructura de datos que alimenta esos reportes: pipelines, lakehouses, transformaciones a escala y gobernanza. Fabric es la plataforma unificada de Microsoft que reemplaza a Azure Synapse + Data Factory + Power BI en un solo lugar.

OneLake = storage único Lakehouse + Warehouse Pipelines = ETL/ELT PySpark + SQL + KQL Delta Lake format Medallion Architecture

🎯 Diferencial clave del examen DP-700

Tenés acceso a Microsoft Learn DURANTE el examen (split screen). Esto no significa que podés buscar todo — el tiempo es ajustado (~2 min por pregunta). Significa que podés consultar sintaxis exacta de KQL/PySpark o nombres de parámetros. Practicá navegar rápido en Learn.

Gestión Microsoft Fabric — arquitectura general 🔥 base de todo
El ecosistema completo que reemplaza a Synapse + ADF + Power BI
▶

Microsoft Fabric es una plataforma SaaS unificada que integra en un solo lugar: Data Engineering (pipelines, Spark), Data Warehousing, Data Science, Real-Time Intelligence, Data Factory y Power BI.

OneLake: El almacenamiento único de Fabric. Similar a OneDrive pero para datos. Todos los items de Fabric (Lakehouse, Warehouse, etc.) guardan sus datos en OneLake automáticamente. Formato subyacente: Delta Parquet.

Items principales de Fabric: Lakehouse · Warehouse · Notebook · Data Pipeline · Dataflow Gen2 · Eventstream · Eventhouse (KQL Database) · Semantic Model · Report

Examen: Fabric es SaaS (Software as a Service), no IaaS ni PaaS puro. No gestionás servidores. La capacidad se mide en CUs (Capacity Units) y se contrata como F2, F4, F8... F64, etc.
Gestión Lakehouse vs Warehouse — cuándo usar cada uno 🔥 muy frecuente
La decisión arquitectónica más importante en Fabric
▶

Lakehouse: Combina data lake (archivos en OneLake) + SQL analytics endpoint. Ideal para datos semiestructurados o sin schema rígido. Se trabaja con PySpark/Spark SQL. Las tablas son Delta tables. Excelente para ETL/ELT y ciencia de datos.

Warehouse: Data warehouse completo con T-SQL nativo. Ideal para datos estructurados con schema definido, consultas complejas con joins, usuarios analíticos que usan SQL. Mayor soporte de concurrencia.

Regla del examen: Datos en múltiples formatos / ciencia de datos / PySpark → Lakehouse. T-SQL completo / alta concurrencia / BI tradicional → Warehouse.
Truco: El Lakehouse tiene un "SQL analytics endpoint" de solo lectura. Para escribir datos via SQL necesitás el Warehouse.
Gestión Medallion Architecture (Bronze → Silver → Gold) 🔥 muy frecuente
El patrón de diseño estándar para lakehouses
▶

Bronze (Raw): Datos tal como llegan de la fuente, sin modificar. Histórico completo. Permite reprocesar si algo falla.

Silver (Cleansed): Datos limpios, validados, deduplicados. Schema aplicado. Unificación de fuentes.

Gold (Curated): Datos listos para consumo: agregados, métricas de negocio, dimensiones. Alimentan reportes Power BI y semantic models.

En Fabric: Podés implementar medallion con carpetas dentro de un Lakehouse o con Lakehouses separados (uno por capa). La segunda opción da mejor aislamiento.
Examen: Si preguntan "minimizar impacto en datos originales" → guardar en Bronze primero. Si preguntan "datos listos para BI" → capa Gold.
Ingestión Pipelines vs Dataflow Gen2 vs Notebook — cuándo usar cada uno 🔥 muy frecuente
Las tres formas de mover y transformar datos en Fabric
▶

Data Pipeline (Data Factory): Orquestación visual de actividades (Copy Data, Notebook, Dataflow, If Condition, ForEach). Ideal para ETL complejos con ramificaciones y dependencias. Sin código.

Dataflow Gen2: Transformaciones con Power Query (M). Interfaz visual tipo Excel. Ideal para usuarios no-técnicos o transformaciones simples/medianas. Escala menos que Spark.

Notebook (PySpark/SparkSQL): Transformaciones complejas con código. Máxima flexibilidad y performance. Ideal para grandes volúmenes o lógica de negocio compleja. Se llaman desde pipelines.

Regla examen: Gran escala + código → Notebook. Sin código + moderado → Dataflow Gen2. Orquestación + dependencias → Pipeline (que puede llamar a Notebooks y Dataflows).
Ingestión Delta Lake — el formato de tabla en Fabric 🔥 crítico
Por qué Delta y qué hace especial este formato
▶

Delta Lake es un formato de tabla open-source (basado en Parquet) que agrega capacidades de base de datos sobre archivos. Todas las tablas en Fabric Lakehouse son Delta tables.

Características clave:

· ACID transactions: Garantiza consistencia aunque múltiples jobs escriban a la vez.

· Time Travel: Podés leer versiones anteriores con VERSION AS OF o TIMESTAMP AS OF.

· Schema enforcement: Rechaza datos que no cumplan el schema.

· OPTIMIZE: Compacta archivos pequeños en archivos más grandes → mejora performance de lectura.

· VACUUM: Elimina archivos viejos que ya no se necesitan (según retention period, default 7 días).

V-Order: Optimización de Fabric sobre Delta Parquet para lectura más rápida en Power BI y SQL. Se aplica automáticamente en Fabric pero podés controlarlo.
Ingestión OneLake Shortcuts 🔥 muy frecuente
Acceso a datos externos sin moverlos
▶

Shortcuts son accesos directos a datos que viven FUERA de Fabric (o en otra parte de Fabric) sin copiarlos. Los datos permanecen en el origen pero aparecen como si estuvieran en el Lakehouse.

Tipos de shortcuts soportados: Azure Data Lake Storage Gen2 · Amazon S3 · Google Cloud Storage · OneLake (otro workspace de Fabric) · Dataverse

Ventaja: Sin duplicar datos. Sin movimiento de datos. El storage del origen sigue siendo la fuente de verdad. Ideal cuando los datos deben quedarse en un sistema existente.
Limitación: Los shortcuts son de solo lectura (no podés escribir al origen via shortcut en la mayoría de casos). Para escribir, necesitás copiar los datos.
Gestión Seguridad en Fabric — capas de acceso 🔥 muy frecuente
Workspace roles + item permissions + OneLake security
▶

Roles de workspace (de mayor a menor permisos): Admin → Member → Contributor → Viewer

Item-level permissions: Permisos adicionales a nivel de item específico (Lakehouse, Warehouse, etc.) para usuarios sin rol en el workspace.

OneLake data access roles: Control granular sobre carpetas dentro de OneLake. Un usuario puede tener acceso solo a ciertas carpetas del Lakehouse.

RLS (Row-Level Security): Restricción de filas en Warehouse con T-SQL o en Semantic Models.

OLS (Object-Level Security): Oculta tablas o columnas completas a ciertos roles.

DDM (Dynamic Data Masking): Enmascara valores sensibles (ej: mostrar "XXXX-1234" en lugar del número de tarjeta real).

Sensitivity labels: Clasifican datos (Confidential, Public, etc.) via Microsoft Purview. Se propagan a los items de Fabric y a los exports.
Optimización OPTIMIZE y VACUUM en Delta tables 🔥 muy frecuente
Performance tuning de tablas Delta
▶
-- Compactar archivos pequeños (mejora performance de lectura) OPTIMIZE lakehouse.schema.tabla -- Con Z-Order (co-localiza datos por columna para filtros más rápidos) OPTIMIZE lakehouse.schema.tabla ZORDER BY (fecha, region) -- Eliminar archivos viejos (default: retiene 7 días de historial) VACUUM lakehouse.schema.tabla -- Reducir retention (CUIDADO: rompe time travel si hay versiones activas) VACUUM lakehouse.schema.tabla RETAIN 168 HOURS
Cuándo correr OPTIMIZE: Después de ingestiones incrementales frecuentes que generan muchos archivos chicos. No es necesario correrlo manualmente si usás Fabric — tiene autocompaction.
VACUUM cuidado: Si reducís el retention a menos de 7 días, perdés la capacidad de hacer time travel a esas versiones. Nunca a 0 horas con procesos activos.
Optimización Capacity Metrics App y throttling ⚡ frecuente
Monitorear el consumo de tu capacidad Fabric
▶

Capacity Metrics App: App de Power BI que muestra el uso de CUs (Capacity Units) en tu tenant de Fabric. Permite identificar qué workloads consumen más y si hay throttling.

Throttling: Cuando una workload consume demasiado, Fabric puede ralentizarla o pausarla para proteger la capacidad. Hay 3 niveles: Interactive delay → Background delay → Rejection.

Examen: Si preguntan "cómo monitorear el consumo de capacidad" → Capacity Metrics App. Si preguntan "por qué los jobs están lentos" → verificar throttling en Capacity Metrics App.
Gestión Git integration y deployment pipelines ⚡ frecuente
ALM y CI/CD en Microsoft Fabric
▶

Git integration: Podés conectar un workspace de Fabric a un repositorio Git (Azure DevOps o GitHub). Los cambios en items se sincronizan. Permite versioning de notebooks, pipelines, dataflows.

Deployment pipelines: Flujo Dev → Test → Prod. Podés promover items entre stages con un botón o automáticamente. Podés configurar reglas por stage (ej: cambiar la fuente de datos entre ambientes).

Examen: Git integration = control de versiones de items. Deployment pipelines = promover entre ambientes. Son dos cosas distintas pero complementarias.

KQL Cheatsheet ⚡

Kusto Query Language — para Real-Time Intelligence en Fabric. Sintaxis clave del examen.

KQL es el lenguaje de consulta para Eventhouses (antes KQL Databases) en Fabric. Se usa para analizar datos de streaming y series temporales. La sintaxis es pipe-based: cada operador transforma el resultado anterior.

Estructura básica y operadores fundamentales
▶
// Estructura: Tabla | operador1 | operador2 | ... MiTabla | where Timestamp > ago(1h) // filtrar por tiempo | where País == "Argentina" // filtrar por valor | project Timestamp, País, Valor // seleccionar columnas | order by Timestamp desc // ordenar | take 100 // limitar filas (= top 100)
ago(): Función de tiempo relativo. ago(1h) = hace 1 hora. ago(7d) = hace 7 días. ago(30m) = hace 30 minutos.
summarize — agregaciones y groupby
▶
// Contar eventos por país MiTabla | summarize Cantidad = count() by País // Suma y promedio MiTabla | summarize TotalVentas = sum(Monto), Promedio = avg(Monto) by Región // Agrupar por intervalo de tiempo (bins) MiTabla | summarize Eventos = count() by bin(Timestamp, 1h) // Contar valores únicos MiTabla | summarize UsuariosUnicos = dcount(UsuarioID) by Fecha
extend, parse, split — columnas calculadas
▶
// extend: agregar columna calculada MiTabla | extend MontoConIVA = Monto * 1.21 | extend EsVIP = iff(Monto > 10000, "VIP", "Normal") // parse: extraer texto estructurado MiTabla | parse Mensaje with "Usuario:" Usuario " Acción:" Accion // tostring / toint / todatetime: conversión de tipos MiTabla | extend FechaReal = todatetime(FechaTexto)
join — combinar tablas en KQL
▶
// Inner join (default) TablaA | join kind=inner (TablaB) on ClaveComun // Left outer join TablaA | join kind=leftouter (TablaB) on ClaveComun // Lookup (más eficiente para tablas de dimensión pequeñas) TablaHechos | lookup TablaClientes on ClienteID
Examen: Preferir lookup cuando la tabla derecha es pequeña (tabla de dimensión). Es más eficiente que join para ese caso.
render — visualizaciones en KQL
▶
// Gráfico de línea temporal MiTabla | summarize Eventos = count() by bin(Timestamp, 1h) | render timechart // Gráfico de barras MiTabla | summarize Total = sum(Monto) by País | render barchart // Pie chart MiTabla | summarize Cantidad = count() by Categoría | render piechart
render siempre va al FINAL del query. Solo funciona en Queryset / KQL queryset, no en pipelines.

📋 Funciones KQL de examen — referencia rápida

count() — total de filas
dcount(col) — distintos (aprox)
avg / sum / min / max
ago(1d) — tiempo relativo
bin(col, 1h) — agrupar tiempo
