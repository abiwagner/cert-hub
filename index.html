ModelarStorage Modes — Import, DirectQuery, Dual🔥 muy frecuente
Lo usás sin saber cómo se llama cada uno
▶

Import: Los datos se copian al modelo. Máxima performance. Permite todas las funciones DAX.

DirectQuery: Power BI consulta la fuente en tiempo real. Siempre actualizado, más lento, limitaciones DAX.

Dual: La tabla puede actuar como Import O DirectQuery según el contexto. Ideal para dimensiones en modelos mixtos.

Regla del examen: "near real time" → DirectQuery. "mejor performance" → Import. Dimensiones en modelo mixto → Dual.
DatosQuery Folding🔥 muy frecuente
El secreto de performance en Power Query
▶

Query Folding es cuando Power Query traduce las transformaciones a código nativo de la fuente (SQL, OData) y deja que la fuente haga el trabajo pesado antes de enviar datos.

Si ocurre → la fuente filtra/ordena/agrupa antes de enviar → menos datos transferidos → más rápido.

Cómo verlo: Clic derecho en un paso en Power Query → "View Native Query". Si está disponible → hay folding. Si está en gris → no hay folding.
Rompe el folding: Columnas personalizadas en M code, combinar fuentes distintas. Los pasos después de uno que rompe el folding tampoco tienen folding.
GestionarRow-Level Security (RLS)🔥 muy frecuente
Seguridad de datos por usuario
▶

Static RLS: Roles con filtros fijos. Ej: País = "Argentina". Se configura en Desktop.

Dynamic RLS: Usa USERPRINCIPALNAME() para comparar con tabla de seguridad. Un solo rol para todos.

FILTER(Tabla, Tabla[Email] = USERPRINCIPALNAME())
Flujo: Crear rol en Desktop → publicar → en Service asignar usuarios al rol. Admins del workspace ven todos los datos sin importar el RLS.
ModelarEsquema estrella vs copo de nieve⚡ frecuente
▶

Estrella: Una tabla de hechos central + tablas de dimensiones desnormalizadas. Power BI está optimizado para esto.

Copo de nieve: Dimensiones normalizadas en sub-dimensiones. Más complejo, menor performance en Power BI.

Siempre elegí estrella en el examen a menos que pregunten específicamente por normalización.
ModelarRelaciones: dirección del filtro cruzado🔥 muy frecuente
▶

Single: El filtro va de la tabla "1" a la tabla "muchos". Default y recomendado.

Both: Filtros en ambas direcciones. Solo para tablas puente en muchos-a-muchos.

Evitar Both en general — puede causar ambigüedad y resultados inesperados.
DatosOn-premises data gateway⚡ frecuente
▶

Necesario cuando la fuente está en la red local y necesitás actualizar desde Power BI Service.

Standard: Compartido, múltiples usuarios y datasets. Personal: Solo un usuario, no recomendado para empresas.

Examen: ¿Cuándo necesitás gateway? Cuando la fuente es on-premises y el reporte está en Service. Sin gateway, el refresh falla.
VisualizarFiltros: jerarquía y tipos⚡ frecuente
▶

1. Filtro de informe: Todas las páginas. 2. Filtro de página: Esa página. 3. Filtro de visual: Solo ese visual. 4. Drill-through: Página de detalle contextual.

Slicer vs Filtro: Slicers son visuales interactivos visibles. Filtros en el panel pueden ocultarse del usuario final.
GestionarWorkspaces y licencias⚡ frecuente
▶

Free: Solo "Mi área de trabajo". No puede compartir. Pro: Workspaces colaborativos. Premium: Usuarios Free pueden consumir contenido de workspaces Premium.

Regla: Usuario A (Pro) comparte con B → B necesita Pro, a menos que el workspace esté en capacidad Premium.

PL-300 · Referencia

DAX Cheatsheet 🔢

Las funciones más frecuentes en el examen con ejemplos.

CALCULATE — La función más importante
Evalúa una expresión en un contexto de filtro modificado
▶
-- Ventas solo de Argentina Ventas AR = CALCULATE(SUM(Ventas[Monto]), Ventas[País] = "Argentina") -- Ventas del año pasado Ventas LY = CALCULATE([Total Ventas], SAMEPERIODLASTYEAR(Fechas[Fecha])) -- Quitar filtro de categoría Ventas All = CALCULATE([Total Ventas], ALL(Productos[Categoría]))
Clave: CALCULATE puede modificar, agregar o quitar filtros del contexto. ALL() dentro quita filtros.
Time Intelligence — YTD, MTD y períodos anteriores
▶
Ventas YTD = TOTALYTD([Total Ventas], Fechas[Fecha]) Ventas MTD = TOTALMTD([Total Ventas], Fechas[Fecha]) Ventas PY = CALCULATE([Total Ventas], SAMEPERIODLASTYEAR(Fechas[Fecha])) Ventas -3M = CALCULATE([Total Ventas], DATEADD(Fechas[Fecha], -3, MONTH))
Requisito: Necesitás tabla de fechas marcada como "fecha" con columna Date sin huecos.
FILTER, ALL, ALLEXCEPT
▶
-- % del total (quitar todos los filtros de la tabla) % Total = DIVIDE([Total Ventas], CALCULATE([Total Ventas], ALL(Ventas))) -- Quitar todos los filtros EXCEPTO País = CALCULATE([Total Ventas], ALLEXCEPT(Ventas, Ventas[País]))
SUMX, AVERAGEX — Iteradores fila por fila
▶
-- SUMX calcula por fila y suma (sin necesitar columna precalculada) Ingresos = SUMX(Ventas, Ventas[Cantidad] * Ventas[PrecioUnitario])
SUM vs SUMX: SUM suma columna existente. SUMX calcula expresión fila por fila y suma. Más flexible pero más lento.
Variables (VAR / RETURN)
▶
Variación % = VAR Actual = [Total Ventas] VAR Anterior = CALCULATE([Total Ventas], SAMEPERIODLASTYEAR(Fechas[Fecha])) RETURN IF(ISBLANK(Anterior), BLANK(), DIVIDE(Actual - Anterior, Anterior))
Las variables se calculan una sola vez — más eficiente y código más legible.

Funciones frecuentes — referencia rápida

DISTINCTCOUNT(col) — valores únicos
DIVIDE(n, d, alt) — división segura
RELATED(col) — columna de tabla relacionada
RANKX(tabla, expr) — ranking dinámico
SELECTEDVALUE(col) — valor único seleccionado
USERPRINCIPALNAME() — usuario actual (RLS)

🆚 DP-700 vs PL-300 — la diferencia clave

El PL-300 es sobre consumir y visualizar datos. El DP-700 es sobre construir la infraestructura que alimenta esos reportes: pipelines, lakehouses, transformaciones a escala y gobernanza. Fabric unifica Azure Synapse + Data Factory + Power BI en un solo lugar.

OneLake = storage único Lakehouse + Warehouse Pipelines = ETL/ELT PySpark + SQL + KQL Delta Lake format Medallion Architecture

🎯 Tip único del DP-700

Tenés acceso a Microsoft Learn DURANTE el examen en split-screen. No significa que podés buscar todo — el tiempo es ajustado (~2 min/pregunta). Usalo para sintaxis exacta de KQL/PySpark o nombres de parámetros. Practicá navegar rápido en Learn.

GestiónMicrosoft Fabric — arquitectura general🔥 base de todo
El ecosistema que reemplaza Synapse + ADF + Power BI
▶

Microsoft Fabric es una plataforma SaaS unificada: Data Engineering, Warehousing, Data Science, Real-Time Intelligence, Data Factory y Power BI en un solo lugar.

OneLake: Storage único de Fabric. Como OneDrive pero para datos. Formato subyacente: Delta Parquet. Todos los items guardan datos aquí automáticamente.

Items principales: Lakehouse · Warehouse · Notebook · Data Pipeline · Dataflow Gen2 · Eventstream · Eventhouse · Semantic Model · Report

Examen: Fabric es SaaS — no gestionás servidores. La capacidad se mide en CUs (F2, F4, F8... F64, etc).
GestiónLakehouse vs Warehouse — cuándo usar cada uno🔥 muy frecuente
▶

Lakehouse: Data lake + SQL analytics endpoint. Datos semiestructurados, schemas flexibles. Se trabaja con PySpark/Spark SQL. Tablas son Delta tables. Para ETL/ELT y ciencia de datos.

Warehouse: T-SQL nativo completo. Datos estructurados con schema rígido. Alta concurrencia. Para BI tradicional y equipos de SQL.

Regla: PySpark / datos variados / ciencia de datos → Lakehouse. T-SQL completo / alta concurrencia / schema rígido → Warehouse.
Truco: El Lakehouse tiene SQL analytics endpoint de SOLO LECTURA. Para escribir via SQL necesitás el Warehouse.
GestiónMedallion Architecture (Bronze → Silver → Gold)🔥 muy frecuente
▶

Bronze (Raw): Datos tal como llegan, sin modificar. Historial completo. Permite reprocesar si algo falla.

Silver (Cleansed): Datos limpios, validados, deduplicados. Schema aplicado.

Gold (Curated): Datos listos para consumo: agregados, métricas, dimensiones. Alimentan Power BI y semantic models.

Examen: "minimizar impacto en datos originales" → Bronze. "datos listos para BI" → Gold.
IngestiónPipelines vs Dataflow Gen2 vs Notebook🔥 muy frecuente
Las 3 formas de mover y transformar datos en Fabric
▶

Data Pipeline: Orquestación visual — encadena Copy Data, Notebooks, Dataflows, If/Else, ForEach. Sin código. Para ETL complejos con dependencias.

Dataflow Gen2: Transformaciones Power Query (M). Visual, sin código. Para transformaciones simples/medianas. Escala menos que Spark.

Notebook (PySpark/SparkSQL): Código. Máxima flexibilidad y performance. Para grandes volúmenes. Se llaman desde Pipelines.

Regla: Gran escala + código → Notebook. Sin código + moderado → Dataflow Gen2. Orquestación con dependencias → Pipeline.
IngestiónDelta Lake — el formato de tabla en Fabric🔥 crítico
▶

Formato open-source (basado en Parquet) con capacidades de base de datos. Todas las tablas en Fabric Lakehouse son Delta tables.

· ACID transactions — consistencia con múltiples writers.
· Time Travel — leer versiones anteriores con VERSION AS OF.
· Schema enforcement — rechaza datos que no cumplan el schema.
· OPTIMIZE — compacta archivos pequeños.
· VACUUM — elimina archivos viejos (default 7 días de historial).

V-Order: Optimización de Fabric sobre Delta Parquet para lectura más rápida en Power BI. Se aplica automáticamente.
IngestiónOneLake Shortcuts🔥 muy frecuente
Acceso a datos externos sin moverlos
▶

Accesos directos a datos fuera de Fabric sin copiarlos. Aparecen como si estuvieran en el Lakehouse.

Tipos soportados: Azure Data Lake Storage Gen2 · Amazon S3 · Google Cloud Storage · OneLake externo · Dataverse

Ventaja: Sin duplicar datos. El storage del origen sigue siendo la fuente de verdad.
Limitación: Los shortcuts son de solo lectura en la mayoría de casos.
GestiónSeguridad en Fabric — capas de acceso🔥 muy frecuente
▶

Workspace roles: Admin → Member → Contributor → Viewer

Item-level permissions: Acceso a un item específico sin rol en el workspace.

RLS: Row-Level Security — restricción de filas.
OLS: Object-Level Security — oculta tablas/columnas.
DDM: Dynamic Data Masking — enmascara valores sensibles.

Sensitivity labels: Clasifican datos via Microsoft Purview y se propagan a exports.
OptimizaciónOPTIMIZE y VACUUM en Delta tables🔥 muy frecuente
▶
-- Compactar archivos pequeños OPTIMIZE lakehouse.schema.tabla -- Z-Order: co-localiza datos por columna (filtros más rápidos) OPTIMIZE lakehouse.schema.tabla ZORDER BY (fecha, region) -- Eliminar archivos viejos (default: 7 días de historial) VACUUM lakehouse.schema.tabla RETAIN 168 HOURS
Cuándo: OPTIMIZE después de ingestiones frecuentes con muchos archivos chicos. Fabric tiene autocompaction pero OPTIMIZE manual da más control.
VACUUM cuidado: Reducir por debajo de 7 días rompe Time Travel de esas versiones.
OptimizaciónCapacity Metrics App y throttling⚡ frecuente
▶

Capacity Metrics App: App de Power BI que muestra el uso de CUs. Permite identificar qué workloads consumen más y si hay throttling.

Throttling niveles: Interactive delay → Background delay → Rejection.

Examen: "cómo monitorear consumo de capacidad" → Capacity Metrics App. "jobs están lentos" → verificar throttling.
GestiónGit integration y deployment pipelines⚡ frecuente
▶

Git integration: Conectar workspace a Azure DevOps o GitHub. Versioning de notebooks, pipelines, dataflows.

Deployment pipelines: Dev → Test → Prod. Promover items entre stages. Configurar reglas por stage (cambiar fuentes de datos entre ambientes).

Examen: Git = control de versiones. Deployment pipelines = promover entre ambientes. Son dos cosas distintas pero complementarias.

KQL Cheatsheet ⚡

Kusto Query Language — para Real-Time Intelligence. Sintaxis pipe-based.

KQL usa pipes (|): cada operador transforma el resultado anterior. Se usa en Eventhouses (KQL Databases) en Fabric para analizar datos de streaming y series temporales.

Estructura básica y operadores fundamentales
▶
MiTabla | where Timestamp > ago(1h) // filtrar por tiempo | where País == "Argentina" // filtrar por valor | project Timestamp, País, Valor // seleccionar columnas | order by Timestamp desc // ordenar | take 100 // limitar filas
ago(): Tiempo relativo. ago(1h)=1 hora. ago(7d)=7 días. ago(30m)=30 minutos.
summarize — agregaciones y groupby
▶
// Contar por país MiTabla | summarize Cantidad = count() by País // Suma y promedio MiTabla | summarize Total = sum(Monto), Prom = avg(Monto) by Región // Agrupar por intervalo de tiempo MiTabla | summarize Eventos = count() by bin(Timestamp, 1h) // Valores únicos aproximados MiTabla | summarize Usuarios = dcount(UsuarioID) by Fecha
extend, iff — columnas calculadas y condicionales
▶
MiTabla | extend MontoConIVA = Monto * 1.21 | extend Segmento = iff(Monto > 10000, "VIP", "Normal") | extend FechaReal = todatetime(FechaTexto)
join y lookup — combinar tablas
▶
// Inner join TablaA | join kind=inner (TablaB) on ClaveComun // Left outer join TablaA | join kind=leftouter (TablaB) on ClaveComun // Lookup (más eficiente para dimensiones pequeñas) TablaHechos | lookup TablaClientes on ClienteID
Preferir lookup cuando la tabla derecha es pequeña (dimensión). Más eficiente que join.
render — visualizaciones inline
▶
// Serie temporal MiTabla | summarize count() by bin(Timestamp, 1h) | render timechart // Barras MiTabla | summarize sum(Monto) by País | render barchart // Pie MiTabla | summarize count() by Categoría | render piechart
render siempre va al FINAL del query.

Funciones KQL frecuentes

count() — total filas
dcount(col) — distintos
ago(1d) — tiempo relativo
bin(col, 1h) — agrupar tiempo
iff(cond, a, b) — condicional
isnotnull(col) — no nulo
startofday/month/year()
strcat(a, b) — concatenar
